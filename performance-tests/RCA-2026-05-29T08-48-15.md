# 14. Incident Runbooks

### Runbook: Authentication Endpoint Returning HTTP 500

- **Trigger:** Alert fires: `error_rate_post_api_login > 50%` OR `http_req_failed rate > 2%` during login phase, OR manual curl to POST /api/login returns 500.

- **Detect:** 
  1. Manually curl the endpoint: `curl -v -X POST http://localhost:3000/api/login -H "Content-Type: application/json" -d '{"username":"test","password":"test"}'`
  2. Confirm response is HTTP 500 (not 503, 502, or 000/timeout).
  3. Repeat 3 times with 2-sec delays to rule out transient glitches. If 2 or more return 500 → proceed to Diagnose.

- **Diagnose:**
  1. **Check service is alive:** `ps aux | grep -E 'node|python|java'` and `netstat -tlnp | grep 3000`. If process is dead, go to Mitigate step 1 (restart).
  2. **Tail error logs (last 50 lines):** `docker logs <container> --tail 50` or `tail -50 /var/log/app.log`. Look for stacktrace, exception type, and the word "JWT" or "sign".
  3. **Check JWT key file:** `ls -la /path/to/jwt/secret` (or equivalent for your environment). Confirm readable and non-empty. If missing → go to Mitigate step 2.
  4. **Verify dependencies:** `curl -v https://<kms-or-auth-provider>/health` (if login depends on external auth). If dependency is down, note it and escalate in parallel.
  5. **Check resource limits (if containerized):** `docker stats <container>` or `kubectl top pod <pod-name>`. If CPU is pinned or memory is near limit → signal resource exhaustion (go to Mitigate step 3).
  6. **Enable debug logging (if available):** Set `LOG_LEVEL=debug` environment variable and restart service. Re-run curl test and capture fresh logs.

- **Mitigate:**
  1. **Restart the service:** `docker restart <container>` or `systemctl restart <app-service>`. Wait for readiness probe to pass (check logs or curl /health). Verify login now works with 3 manual curls.
  2. **If JWT key is missing:** Restore from secure store (e.g., `aws secretsmanager get-secret-value --secret-id jwt-signing-key > /tmp/key.pem` and `mv /tmp/key.pem /path/to/jwt/secret`). Ensure permissions are 600 (`chmod 600 /path/to/jwt/secret`). Restart service.
  3. **If resource limits are breached:** Increase container memory/CPU limit (e.g., from 512 MB to 1 GB) and restart. Monitor for recurrence over next 30 min. If CPU still maxes out, escalate to engineering (code optimization needed).
  4. **If external dependency is down:** Contact the on-call owner of that dependency to restore it. In the meantime, if possible, activate a fallback (e.g., cached token, unauthenticated mode for health checks only).

- **Verify:**
  - Curl POST /api/login 5 times with 1-sec delays. All 5 should return HTTP 200 or 401 (not 500). Latency should be <2 sec.
  - Confirm the k6 test can proceed past the setup phase: run `k6 run script.js --duration=10s --vus=1` and check that login succeeds and a token is acquired.
  - Monitor `error_rate_post_api_login` metric for 5 min. Should remain < 1%.

- **Escalate if:**
  - Restart does not resolve the issue (still 500 after restart + 5 min wait).
  - Logs show a stack trace unrelated to JWT (e.g., database connection error, memory corruption).
  - A critical external dependency (auth provider, KMS) is permanently down; escalate to its team and declare SEV-1 if no ETA for restoration.
  - Resource limits are maxed and increasing them does not help (suggests code defect); escalate to engineering for profiling.

### Runbook: TCP Connection Queueing (http_req_blocked > 10 ms p95)

- **Trigger:** k6 metric `http_req_blocked` p95 exceeds 10 ms OR alert fires for latency percentile anomaly on connection-phase delay.

- **Detect:**
  1. Capture one 30-sec k6 run: `k6 run script.js --duration=30s --vus=5`.
  2. In the summary, find line: `http_req_blocked ......... avg=X.XXms, p(95)=Y.YYms, p(99)=Z.ZZms`. If p95 > 10 ms → proceed to Diagnose.
  3. Compare to baseline: if this is higher than usual, note the change.

- **Diagnose:**
  1. **Check OS-level TCP queue:** `netstat -s | grep -i "listen\|backlog"` or `ss -t -a -m`. Look for lines like "TCP listen queue overflows" or "SYN cookies sent". Non-zero values indicate kernel-level SYN queue exhaustion.
  2. **Check application connection pool (if applicable):** For PostgreSQL, run `SELECT count(*) FROM pg_stat_activity WHERE state = 'idle'`. For MySQL, `SHOW PROCESSLIST`. For Node.js (if using native http), check if the server is accepting new sockets. If idle connections > 100 and wait queue is building, pool is exhausted.
  3. **Measure network round-trip time:** `ping localhost` (should be < 1 ms). If > 2 ms, local network stack is slow.
  4. **Check server listen backlog setting:** `cat /proc/sys/net/ipv4.tcp_max_syn_backlog` (should be ≥ 4096 for typical servers). `cat /proc/sys/net/core/somaxconn` (should be ≥ 1024).
  5. **Check for TIME_WAIT socket buildup:** `ss -t state TIME-WAIT | wc -l`. If > 1000, the service is closing connections faster than TIME_WAIT expires (60 sec default); consider enabling SO_REUSEADDR.

- **Mitigate:**
  1. **Increase listen backlog (if applicable):** `sysctl -w net.ipv4.tcp_max_syn_backlog=8192` and `sysctl -w net.core.somaxconn=8192`. Make persistent by editing `/etc/sysctl.conf`.
  2. **If connection pool is exhausted:** Increase pool size. For HikariCP: set `maximumPoolSize=50` (or 2× your expected concurrent VUs). For pgBouncer: increase `max_client_conn`.
  3. **If server is rejecting connections:** Increase the listen queue size in the application code (e.g., Node.js `server.maxConnections` or Go's `net.ListenConfig.Control` with backlog tuning).
  4. **Monitor TIME_WAIT buildup:** If growing, enable TCP_REUSEADDR in the application (Node.js: `socket.setsockopt(SO_REUSEADDR, 1)`, Python: `socket.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)`).

- **Verify:**
  - Re-run k6 test: `k6 run script.js --duration=30s --vus=5`. Check that `http_req_blocked` p95 < 10 ms.
  - Monitor OS metrics: `netstat -s` should show no new "listen queue overflows".
  - Confirm connection pool is healthy: for PostgreSQL, idle connections should stabilize at ~5–10.

- **Escalate if:**
  - Increasing listen backlog and pool size does not reduce blocking.
  - `netstat -s` still shows large numbers of SYN cookies or listen queue drops.
  - Suggests either: (a) kernel is misconfigured, (b) service has a bug closing connections, or (c) load is genuinely exceeding server's accept capacity at 1 VU (extremely unlikely unless service is single-threaded and CPU-bound). Escalate to infrastructure / platform team and engineering respectively.

### Runbook: Load Test Abort Due to Error Threshold (abortOnFail triggered)

- **Trigger:** k6 test exits early with message: `WARN ... abortOnFail ... threshold on http_req_failed was breached`.

- **Detect:**
  1. Check test output summary for any line containing "abortOnFail" or "threshold ... breached".
  2. Confirm test duration was < expected (e.g., 2 sec instead of 10 min).
  3. Check for metric: `http_req_failed ......... rate=1.00 (2 out of 2 requests failed)`.

- **Diagnose:**
  1. **Identify which endpoint failed:** Look at k6 output for which request group has non-2xx responses. Common causes: login 500, checkout 503, users 502/timeout.
  2. **Count actual vs. expected requests:** If only 2 requests ran, test aborted before scaling. If setup phase failed (login), all endpoints downstream are blocked.
  3. **Check which threshold was breached:** Scan the `thresholds` section of the script. If `http_req_failed rate < 0.02` (2%) was the trigger, then even 1 non-2xx response in a 50-request batch will abort (since 1/50 = 2%).
  4. **Root cause of the specific endpoint failure:** Refer to the endpoint-specific runbooks (e.g., "HTTP 500 on POST /api/login" → use Runbook: Authentication Endpoint, or "HTTP 503 on POST /api/checkout" → escalate to database/lock-saturation runbook once written).

- **Mitigate:**
  1. **Short-term (run test without abort):** Modify script to set `abortOnFail: false` for this one test run to gather data past the first failure. This will show the latency behavior of subsequent endpoints (if any) even if login fails. Example: `thresholds: { 'http_req_failed': [{ threshold: 'rate<0.02', abortOnFail: false }] }`.
  2. **Diagnose the failing endpoint using its dedicated runbook** (see §14 Runbook: Authentication Endpoint if login 500s, or §14 Runbook: Checkout 503 once published).
  3. **Once the endpoint is stable, re-run the test with `abortOnFail: true`** to verify the full load profile completes.

- **Verify:**
  - Run modified script (abortOnFail: false) for 2 min at 5 VUs. Check that all 4 endpoints have at least N=10 requests each and error rates are logged.
  - Restore `abortOnFail: true` and re-run with same parameters. Test should complete full duration without early exit.

- **Escalate if:**
  - Any endpoint still has error rate > 2% even after following the dedicated runbook.
  - Multiple endpoints fail at the same time (suggests resource exhaustion or cascading failure, not isolated endpoint bug).
  - Escalate to service owner and incident commander for deeper investigation.

# 17. Production Readiness Verdict

## **NO-GO** 

The system is **not production-ready** under the tested load profile. The authentication endpoint (critical path) is non-functional at baseline (1 VU, 0 ramp). Gating criteria below must all be satisfied before production deployment.

### Gating Criteria (all must be true)

1. **Baseline authentication endpoint stability:** POST /api/login must return HTTP 200–201 with valid JWT token in response body for 10 consecutive manual curl attempts (1 sec apart) with error rate < 1%. Error rate measured over a 5-min sustained run at 1 VU must be < 0.1%.

2. **p95 latency on login under load:** p95 latency on POST /api/login must be < 2000 ms and p99 < 5000 ms during a 5-min sustained run at target VU count (20 VUs based on script). Must be measured in a staging environment with production-like database and JWT key infrastructure.

3. **Zero dropped iterations during 10-min soak:** Run k6 test with the full 3-stage ramp profile (1 → 5 → 10 → 20 VUs) for 10 min total per stage (30 min cumulative). Measure `dropped_iterations` metric; must be exactly zero (arrival rate not exceeded).

4. **Error rate on all endpoints < 1% sustained:** Across all 4 endpoints (login, checkout, users, health), error rate (non-2xx responses) must remain < 1% during the 30-min ramp test. Specifically:
   - POST /api/login: < 0.1% error rate
   - POST /api/checkout: < 1% error rate (currently expected to hit 503 under load per script comments; must resolve)
   - GET /api/users: < 1% error rate
   - GET /health: < 0.1% error rate

5. **Connection pool and resource headroom confirmed:** Database connection pool must be sized to ≥ 2× expected concurrent VUs (i.e., pool_size ≥ 40 for 20 VUs). Monitor during ramp test: `db_connection_pool_wait_queue_depth` must be zero; if non-zero, increase pool size and re-test.

6. **Observability in place:** Must have (a) server-side error logs with stacktraces retained for ≥ 24 h, (b) application metrics dashboard with JWT signing latency, dependency latency, and connection pool metrics, (c) distributed traces (Jaeger or equivalent) for 100% of login requests. These must be queryable and alertable within 2 min of incidents.

# 18. Post-Incident Review (PIR)

### What went well

- **On-call engineer quickly replicated the issue via curl.** The manual verification (§11 Action #6) provided clear confirmation that the 500 was reproducible and not a transient test artifact, enabling efficient root-cause narrowing.
- **Load test script design was sound.** The 3-stage ramp profile and per-endpoint metrics (duration_*, error_rate_*) provided structure to isolate failures by endpoint. The `abortOnFail` threshold caught the critical failure immediately without wasting test cycles.
- **Root-cause hypotheses were documented in script comments.** The comment "intermittent 500 crashes (JWT signing failure)" in the k6 script directly guided investigation; engineering had already tagged the likely culprit, reducing diagnosis time.

### What went poorly

- **No server-side error logs were provided or captured during the incident.** The analysis (§5, §10) was forced to use [UNKNOWN] and [HYPOTHESIS] labels because the exact exception type, stacktrace, and timestamp were not available. This delayed root-cause confirmation from 5 min (immediate actions) to potentially hours (log capture and audit required).
- **Observability gaps were not pre-existing.** The missing metrics (JWT signing latency, connection pool utilization, dependency response times) identified in §15 should have been instrumented before load testing. Incident response would have been 50% faster with traces and APM telemetry available in real-time.
- **No prior baseline test was run.** The 1-VU baseline test that failed should have been executed in staging days before the formal load test, serving as a smoke test and sanity check. This would have caught the JWT signing failure before the multi-stage ramp test was scheduled.
