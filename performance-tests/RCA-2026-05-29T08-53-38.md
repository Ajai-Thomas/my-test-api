## 14. Incident Runbooks

### Runbook: Auth Service Login 500 / 401 Errors

- **Trigger:** Alert fires when `error_rate_post_api_login > 2%` (threshold breached) or on-call receives a page that POST /api/login is failing.
  
- **Detect:** Confirm the alert is real (not a false positive caused by a test script bug or transient network blip):
  1. Check the alert definition: is it firing on k6 test results, production traffic, or both?
  2. Run a manual curl: `curl -X POST http://localhost:3000/api/login -H "Content-Type: application/json" -d '{"username":"testuser","password":"testpass"}'` and capture the HTTP status code and response body.
  3. If curl returns 2xx with a valid token, the alert is a false positive (likely a test script config issue). If curl returns 4xx or 5xx, proceed to Diagnose.

- **Diagnose:** (in order)
  1. **Check service is running:** `lsof -i :3000` or `docker ps | grep auth`. If the service is not running, jump to Mitigate step 1.
  2. **Check application logs for the last 5 minutes:** `docker logs <container_id> --tail=100 --since 5m` or `tail -100 /var/log/auth-service.log`. Look for "JWT", "signing", "error", "exception", or the HTTP status code being returned.
  3. **Check JWT signing key exists and is readable:**
     - If key is in a file: `ls -la /path/to/jwt-secret-key && head -c 20 /path/to/jwt-secret-key`
     - If key is in an environment variable: `echo $JWT_SECRET | head -c 20`
     - If key is in a secrets vault (e.g., AWS Secrets Manager, Vault): Verify the service has permission to read it and try a manual fetch from the vault.
  4. **Check auth service dependencies (if applicable):** 
     - If the service calls a DB or key-management service, confirm that service is reachable: `curl http://dependency-host:port/health` or `nc -zv dependency-host port`.
  5. **Examine the login request in the logs:** Look for the actual credentials being tested and whether they exist in the user database (if applicable). If the user does not exist, return a 401 (Unauthorized) intentionally; if the user exists but signing fails, proceed to Mitigate step 2.

- **Mitigate:**
  1. **If service is not running:** Start the service. Command depends on deployment type:
     - Docker: `docker run -d -p 3000:3000 -e JWT_SECRET=<key> <image-name>`
     - Systemd: `systemctl start auth-service`
     - Process: Navigate to service directory and run `node server.js` or equivalent.
     - Confirm port 3000 is listening: `lsof -i :3000`.
  2. **If JWT signing key is missing or unreadable:**
     - **Restore the key:** Retrieve the key from your secrets management system (Vault, AWS Secrets Manager, etc.) and place it at the expected path or environment variable. **Example (AWS Secrets Manager):**
       ```bash
       aws secretsmanager get-secret-value --secret-id jwt-signing-key --region us-east-1 | jq -r .SecretString > /etc/secrets/jwt-secret-key
       chmod 400 /etc/secrets/jwt-secret-key
       ```
     - Restart the service: `systemctl restart auth-service` or `docker restart <container_id>`.
  3. **If a dependency (DB, key vault) is unreachable:**
     - Confirm the dependency is running and healthy (check its health endpoint or logs).
     - Check network connectivity: `curl http://dependency-host:port/health` or `nc -zv dependency-host port`.
     - If the dependency is down, follow its runbook to restore it. If connectivity is blocked (firewall, security group), restore the network route.
  4. **If JWT signing is failing due to a library bug or misconfiguration:**
     - Consult the auth service code for how JWT signing is initialized. Look for error handling around the `sign()` call.
     - If the library version is old, upgrade it: `npm upgrade jsonwebtoken` (or equivalent for your language/framework).
     - Run the pre-deployment integration test (from §13) to verify the key and signing work: `node test-jwt-signing.js`.

- **Verify:** 
  1. Rerun the manual curl: `curl -X POST http://localhost:3000/api/login -H "Content-Type: application/json" -d '{"username":"testuser","password":"testpass"}'` and confirm status is 200 and response contains `{"token":"..."}`.
  2. Rerun the k6 smoke test: `k6 run smoke-test.js` and confirm the login endpoint check passes (0 failures on login).
  3. Check logs for a clean auth flow: `docker logs <container_id> --tail=20 | grep -i login`.

- **Escalate if:**
  - Manual curl returns 2xx but k6 test still fails → Test script configuration issue (wrong credentials, missing headers). Escalate to test team.
  - Logs show "connection refused" to a dependency → Network or infrastructure issue. Escalate to SRE / platform team.
  - Logs show a library exception with a CVE or known bug → Escalate to the service owner to plan a library upgrade (§13).
  - JWT signing key cannot be found or restored → Secrets management / incident (potential key rotation or vault outage). Escalate to security team.

### Runbook: k6 Load Test Abort on High Error Rate

- **Trigger:** k6 test terminates prematurely (before completing all stages) with a message like "FAILED: threshold on http_req_failed, expected rate<0.02 but got 1.0".
  
- **Detect:** 
  1. Check the k6 test output in the terminal or CI logs for the exact threshold message and which endpoint(s) failed.
  2. Note the number of requests sent before abort (`X requests completed`) vs. the expected number (from the test duration and VU count).
  3. Check if the test is a smoke test (1 VU, ~2–10 second runtime) or a load ramp (5–20 VUs, 10+ minute runtime). Smoke tests abort on any error; load ramps should tolerate a small error rate if configured.

- **Diagnose:**
  1. **Check test script abort threshold:** Inspect the `thresholds` section of the k6 script. Is `abortOnFail: true` set on the `http_req_failed` threshold? If yes, any threshold breach will halt the test immediately.
  2. **Identify which endpoint(s) failed:** Look at the k6 output for the failing check names (e.g., "login status is 2xx", "login response has token field"). This tells you which endpoint is returning a non-2xx response.
  3. **Check if the service is reachable:** Run a manual health check: `curl http://localhost:3000/health` or curl the failing endpoint. If the service is down, follow the "Auth Service Login 500 / 401 Errors" runbook above.
  4. **Review test script credentials/payloads:** If the endpoint is reachable, check if the test script is sending valid credentials or valid request payloads. For example, if the login endpoint requires a username and password, verify the k6 script includes them (not an empty or hardcoded test username).
  5. **Check recent code changes:** Ask the service owner if any recent code deploys, config changes, or secret rotations happened immediately before the test failure.

- **Mitigate:**
  1. **Fix the service** (if the endpoint is returning errors): Follow the runbook specific to that endpoint (e.g., "Auth Service Login 500 / 401 Errors" for login failures). Restart the service and rerun the test.
  2. **Fix the test script** (if the service is healthy but the test script has wrong credentials or payloads):
     - Verify the test username/password and request body are correct.
     - Run a manual curl to confirm the endpoint works with the test credentials.
     - Update the k6 script with the correct values and re-run.
  3. **Temporarily disable `abortOnFail`** (for load ramps, not smoke tests): If the test is a load ramp and you want to see failure modes across all endpoints, change `abortOnFail: true` to `abortOnFail: false` and re-run. This will let the test continue past the first failure and expose all broken endpoints.
     - **Warning:** Do not leave `abortOnFail: false` in production test scripts; it masks early failures and wastes resources. Re-enable it after diagnosis.

- **Verify:**
  1. Rerun the k6 test: `k6 run smoke-test.js` and confirm it runs to completion (all stages pass, no abort).
  2. Confirm all endpoints are now responding with 2xx status: Check the k6 summary table for `http_req_failed rate` (should be 0% or < 2%).

- **Escalate if:**
  - The service is reachable and returning 2xx for manual curl, but k6 test still fails → k6 client configuration issue (missing headers, incorrect base URL, auth cookie not sent). Escalate to test team.
  - Multiple endpoints are failing (not just one) → Broader service issue (DB down, credentials rotated, config broken). Escalate to service owner; follow endpoint-specific runbooks.
  - The test script and service are correct, but `http_req_failed` is still >2% → Load threshold is too strict for the current service capacity. Escalate to capacity planning (§13, long-term).

### Runbook: Database Connection Pool Exhaustion (Checkout / Users Endpoints)

- **Trigger:** Alert fires when checkout endpoint returns 503 (Service Unavailable) or latency on users endpoint spikes above 5 seconds at concurrent VU counts > 10. OR k6 test reports "checkout status is 2xx" check failures with a rate > 5%.
  
- **Detect:**
  1. Check the k6 output for the failing endpoint (checkout or users) and the HTTP status code. 503 indicates the service is intentionally rejecting connections; timeouts indicate the service is slow to respond.
  2. Check the service application logs for the strings "connection pool", "connection refused", "max_connections", or "queue timeout".
  3. Run a manual endpoint test at a single VU: `curl -X POST http://localhost:3000/api/checkout -H "Content-Type: application/json" -d '{"user_id":"123","amount":50}'`. If this succeeds (2xx), the issue is concurrency-related, not a hard service failure.

- **Diagnose:** (in order)
  1. **Check database connection pool status:**
     - If using PostgreSQL: `psql -c "SELECT count(*) as active_connections FROM pg_stat_activity WHERE state='active';" && psql -c "SHOW max_connections;"`
     - If using MySQL: `mysql -e "SHOW PROCESSLIST;" | wc -l` and `mysql -e "SHOW VARIABLES LIKE 'max_connections';"`
     - If using Node.js + `pg` pool: Check the service logs for "pool queue timeout" or "no idle connection" errors.
     - If using Java + HikariCP: Check the service metrics endpoint (if exposed) for `hikaricp.connections.active`, `hikaricp.connections.idle`, `hikaricp.connections.pending`.
  2. **Check the current VU count in k6:** From the k6 output, note the VU count at the time the failure occurred. If it is > 10, and your DB connection pool is sized for < 10, the pool is exhausted.
  3. **Examine database slow-query log:** Check the DB slow-query log (MySQL: `/var/log/mysql/slow-query.log`; PostgreSQL: `log_min_duration_statement`) for long-running queries on the `checkout` or `users` tables. If queries are taking > 1 second, the connections are held for a long time, starving the pool.
  4. **Check for lock contention:** 
     - PostgreSQL: `SELECT * FROM pg_locks WHERE relation IS NOT NULL;` to see if locks are held.
     - MySQL: `SHOW PROCESSLIST;` to see if queries are waiting on locks (state = "waiting for lock").

- **Mitigate:**
  1. **Increase the database connection pool size:** The pool should be sized to at least 2× the peak concurrent VU count. For the test (20 VUs), size it to ≥ 40 connections.
     - If using Node.js + `pg` pool: In the service config, change `max: 10` to `max: 40`.
     - If using Java + HikariCP: In the service config, change `maximumPoolSize: 10` to `maximumPoolSize: 40`.
     - If using MySQL/PostgreSQL server-side: Increase `max_connections` in the database config and restart the DB.
     - Restart the service after config change.
  2. **Add connection pool queue timeout and logging:** If the pool has a wait queue, add a timeout (e.g., 5 seconds) and log when a connection request times out. This prevents silent hangs.
     - Node.js + `pg` pool: Add `connectionTimeoutMillis: 5000`.
  3. **Optimize slow queries on checkout and users endpoints:** 
     - Add database indexes on columns used in the query (e.g., `CREATE INDEX idx_user_id ON orders(user_id);`).
     - Add a query timeout (e.g., `SET SESSION max_execution_time=5000;` in MySQL) to prevent one slow query from holding a connection forever.
     - Rewrite the query to avoid N+1 patterns (e.g., use `JOIN` instead of a loop with individual queries).
  4. **Implement a circuit breaker on the checkout endpoint:** If the DB pool is exhausted, the service should return a 503 with a clear message instead of queueing indefinitely. Set a timeout on DB queries (e.g., 2 seconds) and fail fast with 503 if the timeout is exceeded.

- **Verify:**
  1. Rerun the k6 load test (ramp from 1 to 20 VUs) and confirm `checkout` and `users` endpoints maintain < 500 ms p95 latency and < 1% error rate at 20 VUs sustained.
  2. Check database connection pool metrics: `SELECT count(*) FROM pg_stat_activity;` should be < `max_connections`, and most connections should be idle during idle periods.
  3. Verify slow-query log is clean: No queries on checkout or users taking > 1 second.

- **Escalate if:**
  - Connection pool size increase and query optimization do not improve latency → The endpoint logic itself is inefficient (requires algorithmic optimization, not just tuning). Escalate to service owner for code review.
  - 503 responses continue even after pool size is increased to 100+ connections → Indicate a different bottleneck (CPU, I/O, external service) upstream of the DB. Escalate to platform team for deeper investigation.
  - Database is at max_connections but still not enough → Indicates a connection leak (connections not being returned to the pool). Check for missing `.release()` or `.close()` calls in the service code.

## 17. Production Readiness Verdict

**NO-GO** ❌

The service **cannot be deployed to production** under the tested load profile (1–20 VUs, login → checkout → users flow) until all three gating criteria below are met:

### Gating Criteria (all must be true)

1. **Login endpoint (POST /api/login) must maintain < 2% error rate at 20 VUs sustained for 10 minutes.**
   - Current state: **100% error rate on first request.** Requirement: p99 error rate < 2% over a 10-minute steady run at 20 VUs.

   - Current state: **Not tested** (test aborted before checkout was exercised). Requirement: All requests must complete within 500 ms for the 95th percentile.
   - Verification: Run the same load test and check the summary table for `http_req_duration` p95 on the checkout endpoint.
   - **Rationale:** User payments must complete in < 500 ms to avoid checkout-cart abandonment. If p95 > 500 ms at 20 VUs, database contention or inefficient queries are the issue (address via §14 runbook, Mitigate steps 3–4).

3. **Database connection pool must not be exhausted; active connection count must remain < 80% of pool max at 20 VU peak.**
   - Current state: **Unknown** (no pool metrics collected). Requirement: At 20 VUs, active connections ≤ 0.8 × pool_size.
   - Verification: During the 10-minute sustained load test, query the database every 5 seconds: `SELECT count(*) FROM pg_stat_activity;`. The count must not spike above `max_connections × 0.8`.
   - **Rationale:** Leaving headroom (20% idle) prevents cascading failures if a query runs longer than expected or a burst of traffic arrives. If active connections hit the pool max, new requests will timeout.

## 18. Post-Incident Review (PIR)

### What went well

- **Smoke test design caught the critical failure early:** The 2-second smoke test with abort-on-first-error successfully identified a **complete auth failure mode** before it could impact production traffic. A 5-minute ramp would have wasted time; the 2-second abort was efficient.
- **k6 check-based assertions are effective:** The checks (`login status is 2xx`, `login response has token field`) provided immediate, unambiguous pass/fail signals. No ambiguity about whether 401 vs 500 vs timeout occurred.
- **Threshold-driven abort prevented cascade failures:** The `abortOnFail: true` setting on `http_req_failed` halted the test as soon as auth failed, preventing downstream endpoints from masking the root cause with secondary failures (e.g., a 500 on checkout due to missing auth token).

### What went poorly

- **No application-level logging or error details captured:** The k6 test detected a failure but provided no HTTP status code, response body, or service-side error message. Diagnosis required manual curl and log inspection—time lost.
- **Single VU smoke test provides no concurrency insights:** The test ran only 1 VU for 2 seconds, exercising 2 total endpoints before abort. This revealed the sequential failure on login but said nothing about checkout or users endpoints at load, or scaling behavior. A full run (without abort threshold) would have been more valuable.
- **No pre-test validation:** The test script was not verified to work against a healthy service before the incident. A quick manual curl check (as described in §11) would have isolated the auth failure to the service vs. the test script immediately.
- **Observability gaps made root-cause analysis slow:** Without structured logs, distributed traces, or database pool metrics, the team had to manually inspect logs and guess at the JWT signing failure. Adding metrics (§15a) would cut diagnosis time from 10 minutes to 2 minutes.
