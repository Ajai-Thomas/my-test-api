# 14. Incident Runbooks

### Runbook: **k6 Smoke Test Setup Failure (Auth 401)**

- **Trigger:** k6 smoke test aborts after <5 requests with error message `[setup] login returned 401 — running unauthenticated` in stdout AND `http_req_failed` threshold breached (>2% error rate).

- **Detect:** 
  1. Grep k6 stdout for the literal string `login returned 401`.
  2. Confirm that `http_req_failed` rate metric reached 100% (all requests returned 4xx or 5xx).
  3. Verify that the test completed <30 seconds of execution and <20 requests total (indicating early abort, not a long-running failure).

- **Diagnose:** 
  1. Check environment variables: `echo "BASE_URL=$BASE_URL USERNAME=$USERNAME PASSWORD=$PASSWORD"`. If any are empty or obviously wrong (e.g., USERNAME=test but that's not a valid user), proceed to credential correction.
  2. Curl the login endpoint directly (outside k6): `curl -X POST $BASE_URL/api/login -H 'Content-Type: application/json' -d '{"username":"'$USERNAME'","password":"'$PASSWORD'"}'`. Inspect the response: should be 200 with a token field, or 401 with an error message. If 401, the credentials are wrong; if 200, the script setup logic is broken (token extraction or header format).
  3. If curl returns 200, re-run k6 with debug logging: `k6 run -v --logformat=json script.js 2>&1 | jq '.message | select(. | contains("login") or contains("token") or contains("auth"))'` to see what the setup function is doing with the response.
  4. Check the login response body format: `curl -X POST $BASE_URL/api/login -H 'Content-Type: application/json' -d '{"username":"'$USERNAME'","password":"'$PASSWORD'"}' | jq .` to see the exact field names (may be `token`, `access_token`, `jwt`, `data.token`, etc.).

- **Mitigate:** 
  1. **If credentials are wrong:** Obtain valid credentials from the service owner or auth system (LDAP, OAuth provider, or user database). Update the environment variables: `export USERNAME=<valid_user> PASSWORD=<valid_pass>` and re-run the test.
  2. **If token extraction is broken:** Edit the k6 script's setup function. Locate the line `token = res.json('token') || ''`. Change `'token'` to match the actual field name from the login response (e.g., `res.json('access_token')` or `res.json('data.token')`). Re-run the test.

### Runbook: **k6 High Blocking Latency (TCP Connection Pool Starvation)**

- **Trigger:** k6 test completes successfully but `http_req_blocked` p95 > 10 ms OR `http_req_blocked` average is >5 ms AND `http_req_connecting` average is >0.1 ms (indicating repeated TCP handshakes).

- **Detect:** 
  1. Run k6 test to completion and capture stdout.
  2. Search for the `http_req_blocked` metric in the summary (format: `http_req_blocked {avg: X.XXms, p95: Y.YYms, p99: Z.ZZms, ...}`).
  3. If p95 > 10 ms OR avg > 5 ms, this runbook is triggered.
  4. Also check `http_req_connecting` avg; if >0.1 ms, it suggests new TCP connections are being created per request rather than reused.

- **Diagnose:** 
  1. Review the k6 script configuration: search for `jar:`, `Connection:`, or connection pool settings. By default, k6 may not enable HTTP/1.1 keep-alive.
  2. Run the test with explicit keep-alive enabled: add `export K6_HTTP_CONN_REUSE_KEEP_ALIVE=true` before running k6, then re-run and compare `http_req_blocked` metrics.
  3. Check the target service HTTP response headers for `Connection: keep-alive`. Run: `curl -i http://localhost:3000/api/health | grep -i connection`. Should see `Connection: keep-alive` in the response.
  4. If the target service is behind a load balancer (ALB, NGINX, etc.), verify that the load balancer is configured to forward keep-alive headers and not closing connections prematurely. (Confirm with service owner if topology is unknown.)

- **Mitigate:** 
  1. **If keep-alive is not enabled in k6:** Add to the k6 script top-level:
     ```javascript
     export const options = {
       // ... other options ...
       ext: {
         loadimpact: {
           // projectID: 1234,
         }
       }
     };
     ```
     Or set the environment variable before running: `k6 run --vus 1 --duration 30s script.js` with `K6_HTTP_CONN_REUSE_KEEP_ALIVE=true` in the shell.
  2. **If the target service does not support keep-alive:** Escalate to the service owner; request that HTTP/1.1 keep-alive be enabled in the application or load balancer config.
  3. Repeat the test and collect new `http_req_blocked` metrics.

- **Verify:** 
  - `http_req_blocked` p95 should drop by ≥ 30% (e.g., from 15 ms to 10 ms or lower).
  - `http_req_connecting` average should drop toward 0 (no new TCP handshakes).
  - Overall `http_req_duration` should improve as a result.

- **Escalate if:** 
  - After enabling keep-alive, `http_req_blocked` p95 remains >10 ms: the bottleneck may be application-layer request queueing (e.g., slow request handlers blocking new requests). Escalate to §15 (Observability Gaps) and request server-side metrics (CPU, request queue depth, handler latency). Or investigate k6 script concurrency: if the script is single-threaded or has synchronous waits, increasing VU count may help.
  - Target service does not respond with `Connection: keep-alive`: it may be misconfigured or behind a proxy that strips the header. Escalate to service owner.

### Runbook: **k6 Test Abort on http_req_failed Threshold (Production-Readiness Gate)**

- **Trigger:** k6 test stops prematurely with message `thresholds on metrics 'http_req_failed' were crossed; at least one has abortOnFail enabled, stopping test prematurely` AND test completed <10% of the planned duration or <100 total requests.

- **Detect:** 
  1. Check k6 stdout for the literal string `abortOnFail`.
  2. Count total requests in summary (`http_reqs count = N`). If N < 100, the test was cut short.
  3. Verify that the abort was due to `http_req_failed` rate, not other thresholds (check the threshold definition in the script options: should show `http_req_failed: [{ threshold: 'rate<0.02', abortOnFail: true }]`).

- **Diagnose:** 
  1. Identify the HTTP status codes in the test run. Search k6 stdout for lines with `http_req_status {200: X, 401: Y, 500: Z, ...}`. Note which status codes are failing (typically 4xx or 5xx).
  2. If 401 or 403 is the dominant failure code, this is an authentication issue—use the Auth 401 runbook (above).
  3. If 500 or 5xx is dominant, the service is returning errors: check service logs, metrics, and error rate. Is the service crashing, out of memory, or returning unhandled exceptions?
  4. If mixed status codes (e.g., 200, 201, 202, 400, 401), determine which endpoint is failing. Run: `k6 run --summary-export=summary.json script.js 2>&1 > test.log && jq '.metrics.http_req_status | to_entries | map({status: .key, count: .value.value})' summary.json` to break down failures by status per endpoint (if the script exports per-endpoint metrics).
  5. Calculate failure rate: `failed_requests / total_requests`. If >2% (the threshold), the test is correctly aborting; the question is why the failure rate is high. Possible causes: (a) auth issue (see runbook above), (b) endpoint does not exist or has been removed, (c) service is overloaded and returning 503 errors, (d) downstream dependency (database, external API) is down.

- **Mitigate:** 
  1. **If auth is the issue:** Use the Auth 401 runbook.
  2. **If the endpoint does not exist or is misconfigured in the script:** Verify the endpoint URL in the script against the API spec. Correct any typos (e.g., `/api/users` vs `/api/user`, or missing path parameters). Re-run.
  3. **If the service is returning 5xx errors:** Check service logs and metrics. Look for recent deployments, database connection failures, out-of-memory, or cascading failures from upstream services. Restart the service or rollback the recent deployment if needed.
  4. **If failures are due to load (service overloaded and returning 503):** This suggests the test profile (VU count, request rate) is beyond the service's capacity. This is expected behavior for a smoke test and is not a code defect—it is a capacity planning finding. Document the breaking point and discuss with the service owner.
  5. If all of the above are ruled out, lower the `abortOnFail` threshold temporarily to allow the test to continue running: change `abortOnFail: true` to `abortOnFail: false` in the script. This will allow the test to run to completion even if failures exceed 2%, providing more data for diagnosis. Caution: only use this in non-production runs; in CI/CD, a high failure rate should still block deployment.

- **Verify:** 
  - Test should run to completion (not abort early).
  - `http_req_failed` rate should be <2% and not trigger the abort.
  - Summary should show at least 100 requests and data from multiple VU levels (if the test includes ramp-up).

- **Escalate if:** 
  - Test completes but error rate is 2–10%: service is partially degraded. Escalate to service owner with logs and metrics.
  - Error rate is 10–30%: service is significantly degraded or under extreme load. Escalate to on-call engineer and consider rolling back recent changes.
  - Error rate is >30% or 100%: service is down or unreachable. Page on-call and escalate to war room.

# 18. Post-Incident Review (PIR)

### What went well

- **Test environment was available and accessible.** The smoke test script executed and produced diagnostic output (login check passed, latency metrics recorded) despite the authentication failure. This enabled quick root-cause isolation to the setup phase.
- **Thresholds were appropriately configured for a smoke test.** The `abortOnFail` threshold on `http_req_failed` < 2% correctly triggered an early abort when the test encountered 100% failure rate, preventing wasted compute and time on a fundamentally broken test run.
- **Clear logging of setup failure.** The warning message `[setup] login returned 401 — running unauthenticated` was explicit enough to identify the root cause within minutes of reviewing the logs.

### What went poorly

- **Test setup phase did not validate success before proceeding.** The setup function logged a warning (401) but did not throw an error or halt the test. The script continued to send unauthenticated requests, which predictably failed, wasting cycles and potentially confusing the test output with service failures.
- **Credentials were either missing or incorrect in the test environment.** The root cause (invalid username/password or missing environment variables) was not caught before the test run. No pre-flight validation checked that auth credentials were valid.
- **Single-run sample (N=2 requests) is insufficient for any capacity or reliability conclusion.** The test aborted so early that no meaningful latency or throughput data was collected. A single-run abort does not provide actionable feedback on whether the service is ready for production load.
- **No observability for TCP-level performance.** The latency decomposition metrics (http_req_blocked, http_req_waiting breakdown) were present in the output but were not visualized or analyzed. The dominance of `http_req_blocked` (7.95 ms avg, p95 14.61 ms) relative to total latency (3.61 ms avg) should have triggered immediate investigation into connection pooling, but this was not done proactively.
