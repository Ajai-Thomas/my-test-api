### Runbook: Service Responds with HTTP 500 to All Requests

- **Trigger:** k6 load test fails during setup/ramp phase; stdout log shows "login returned 500" or similar; any manual test of `curl http://localhost:3000/api/<endpoint>` returns HTTP 500 for all endpoints.

- **Detect:** 
  1. Run `curl -v http://localhost:3000/api/health` and check the HTTP status code in the response line (e.g., "< HTTP/1.1 500 Internal Server Error").
  2. Check the service process: `ps aux | grep <service_name_or_port_pattern>`. If the process is not listed, the service is not running.
  3. If the process exists, tail the application error log (confirm location with service owner; typical paths: `/var/log/app.log`, `~/.local/share/app/logs`, or `docker logs <container_id>`). Search for "error", "exception", or "500" in the past 5 minutes.

- **Diagnose:**
  1. **Is the service process running?** If no, go to *Mitigate* step 1. If yes, continue.
  2. **Is the service listening on port 3000?** Run `netstat -tlnp | grep 3000` (Linux) or `netstat -ano | findstr 3000` (Windows). If port is not in LISTEN state, the service crashed or failed to bind. Restart and go to *Mitigate* step 1.
  3. **Are dependencies available?** Ask the service owner: "Does this service depend on a database, cache, or external API?" If yes, verify connectivity: `telnet <dependency_host> <port>` (or equivalent for HTTP endpoints: `curl -I http://<dependency_url>`). If connectivity fails, that is the root cause.
  4. **Are environment variables set?** Run `env | grep -E "DB|AUTH|API"` or equivalent, specific to the service. If required variables are missing, that is the root cause.
  5. **What does the application log say?** Grep the log file for the past 5 minutes: `tail -100 /path/to/app.log | grep -E "error|exception|fatal"`. If logs show a database connection error, auth timeout, or uncaught exception, that is the root cause.

- **Mitigate:**
  1. **If service is not running:** Restart the service using the documented startup command (e.g., `systemctl restart my-service` or `docker-compose up -d service-name` or equivalent). Wait 5 seconds, then run `curl http://localhost:3000/api/health` to confirm it is responsive.
  2. **If dependencies are unavailable:** Contact the dependency owner (e.g., database team, auth team) and ask them to confirm status. Do not restart the service until dependencies are verified online. Once confirmed, restart the service.
  3. **If environment variables are missing:** Set the required variables in the service's configuration file or startup script (confirm location with service owner). Restart the service. Verify via `env | grep <VAR>` that the variable is now set.
  4. **If application logs show an uncaught exception:** File a P1 bug with the service owner and include the full exception stack trace from the log. Do not attempt to fix code in production; escalate to the development team.

- **Verify:**
  1. Run `curl -w "\n%{http_code}\n" http://localhost:3000/api/health` and confirm the HTTP status is 200.
  2. Run the smoke test: `k6 run --vus 1 --duration 2m load-test.js`. The test should complete without aborting on threshold failures. All requests in the setup phase should return 2xx.
  3. Check the service logs: `tail -20 /path/to/app.log` should show no new errors or warnings in the past 2 minutes.

- **Escalate if:**
  - The service restarts successfully but returns 500 again within 5 minutes (indicates a transient or recurring issue, not a static misconfiguration).
  - The dependency is confirmed online, but the service still returns 500 (indicates an application bug, not an infrastructure issue).
  - The service logs do not provide a clear error message (escalate to the development team with the logs and ask them to add more verbose logging).
  - **Escalate to:** Service owner / on-call engineer + development team lead if root cause is an uncaught exception.

### Runbook: High http_req_blocked Metric / Connection Pool Saturation

- **Trigger:** k6 reports `http_req_blocked` p95 > 50 ms, or `http_req_blocked` > total `http_req_duration` (metric decomposition violation). This occurs when requests are queued waiting for a connection from the pool.

- **Detect:**
  1. Check the k6 summary output for the metric `http_req_blocked` and its p95 value. If p95 > 50 ms or if the avg exceeds the total `http_req_duration`, connection queuing is present.
  2. Alternatively, enable k6's verbose HTTP logging: re-run the test with `--http-debug=full` and search the logs for "connection pool exhausted" or "waiting for connection" messages.
  3. On the target service host, run OS-level diagnostics: `ss -s` (Linux) to check socket counts, or `netstat -an | grep ESTABLISHED | wc -l` to count active connections. Compare against the k6 VU count; if active connections >> VUs, connection reuse is failing.

- **Diagnose:**
  1. **Is the connection pool size sufficient?** Confirm with the service owner: "What is the maximum concurrent HTTP connections the service can accept?" If the service is deployed on a single process (e.g., Node.js single-threaded server), the limit may be low. Check the service config for `maxConnections`, `backlog`, or `ulimit` settings.
  2. **Is k6's connection pool sized appropriately?** In the k6 script, check the `http.Client()` configuration or the global `http` module settings. By default, k6 reuses connections (keep-alive), but if the service is closing connections, the pool may be undersized. Confirm: Is the target service sending `Connection: close` headers, or is it properly supporting `Connection: keep-alive`?
  3. **Are there network-level bottlenecks (DNS, firewall)?** Run `dig localhost` or `nslookup localhost` to confirm DNS resolution is fast. If localhost is resolving via a remote DNS server instead of the loopback resolver, latency will spike. On the target host, check firewall rules: `iptables -vnL` (Linux) or `netsh advfirewall show allprofiles` (Windows) to ensure port 3000 is not rate-limited or queued.

- **Mitigate:**
  1. **Increase the service's connection limit:** If the service is a Node.js app, set `server.maxConnections = <desired_limit>` (e.g., 1000 for 20 VUs with 10 connections each). Restart the service.
  2. **Disable DNS lookups in k6:** If the target is localhost, hardcode the IP in the k6 script: `http.get('http://127.0.0.1:3000/api/...')` instead of `http://localhost:3000/...`. This eliminates DNS resolution latency.
  3. **Verify keep-alive is enabled:** In the k6 script, ensure `http.KeepAlive: true` is set (it is by default). If the service is closing connections, ask the service owner to enable keep-alive in the HTTP server config.
  4. **Reduce VU count or add think time:** If the bottleneck cannot be resolved with infrastructure changes, reduce the number of concurrent VUs in the load test or add `sleep(1)` between requests to lower the request arrival rate.

- **Verify:**
  1. Re-run the load test with the same VU profile. Check the new `http_req_blocked` p95 value. It should be < 50 ms and should not exceed total `http_req_duration`.
  2. Run `ss -s` or `netstat -an | grep ESTABLISHED | wc -l` on the service host during the test. Active connections should be ≤ 2 × VU count (allowing for keep-alive connection reuse).
  3. Check k6 output: `http_req_blocked` should be dominated by `http_req_connecting` (TCP handshake) and `http_req_waiting` (TTFB), not queuing delay.

- **Escalate if:**
  - `http_req_blocked` remains high after increasing the service's connection limit.
  - OS-level diagnostics show that the kernel's socket backlog is full (`netstat -s | grep "segments retransmitted"`).
  - **Escalate to:** Infrastructure team / service owner + network team for firewall audit and connection limit tuning.

### Runbook: Load Test Aborts on Error Rate Threshold

- **Trigger:** k6 test aborts mid-run with message "Threshold reached on group 'post_api_login'", error_rate_post_api_login breach, or similar. Test does not complete full load profile (ramp, sustain, ramp-down).

- **Detect:**
  1. Check the k6 console output for the line `Group <name> abort: threshold`.
  2. Check the thresholds section of the k6 script (in the input, `options.thresholds`): is there a `rate(...)` threshold with `abortOnFail` set to true for the failing endpoint?
  3. Manually test the endpoint that caused the abort: `curl -w "\n%{http_code}\n" http://localhost:3000/api/<endpoint>` and count how many requests fail. If > 5%, the threshold is being breached.

- **Diagnose:**
  1. **Is the error a transient failure or a persistent one?** Run the curl command 10 times and count successes vs. failures. If all 10 fail, the issue is persistent (go to *Mitigate* step 1). If some succeed, the issue is transient (go to *Mitigate* step 2).
  2. **Does the error occur only under load?** Run the smoke test with 1 VU: does the endpoint respond successfully? If yes, this is a load-induced failure (connection saturation, resource exhaustion). If no, this is a baseline service issue (follow the "Service Responds with HTTP 500" runbook).
  3. **Are upstream services or databases overloaded?** If the service depends on a downstream backend, check that backend's metrics (CPU, memory, connection count, query latency). High tail latency on the backend may cause the service to timeout and return 500 errors.

- **Mitigate:**
  1. **If the error is persistent:** Follow the "Service Responds with HTTP 500 to All Requests" runbook to fix the service.
  2. **If the error is transient and only under load:** Either (a) increase the error rate threshold in the k6 script (change `rate(0.05)` to `rate(0.10)`, for example), or (b) increase the service's resource capacity (add more VMs, scale up the database, add read replicas) to handle the load. Option (b) is preferred; option (a) is a temporary workaround.
  3. **Remove or adjust abortOnFail:** If the threshold breach is expected (e.g., 10% error rate at 100 VUs is acceptable), set `abortOnFail: false` for that threshold in the k6 script. The test will continue and collect data even if the threshold is breached; this allows you to see the full load profile and tail-latency percentiles.

- **Verify:**
  1. Re-run the load test. It should not abort before the ramp-down phase.
  2. Check the final summary: all four endpoints should have been exercised. If any endpoint shows "0 requests", it was not reached due to an earlier abort.
  3. Review the thresholds section of the k6 summary: confirm that error_rate thresholds either pass or fail within acceptable bounds (do not exceed SLO).

- **Escalate if:**
  - The error rate exceeds 50% at steady-state (20 VUs), indicating a fundamental scaling issue.
  - The error is intermittent and cannot be reproduced manually (suggests a race condition or timing-dependent bug).
  - **Escalate to:** Service owner / on-call engineer + development team if the issue is application-level.

### Runbook: Test Data / Credentials Invalid for Load Test

- **Trigger:** k6 setup phase completes but logs show "login returned 401" or "authentication failed" instead of "login returned 500". Test continues with a note like "running unauthenticated". Later stages return 401/403 for protected endpoints.

- **Detect:**
  1. Check the setup phase logs in the k6 output: if the login status is 401, 403, or 422 (unprocessable entity), credentials are the issue.
  2. Manually test: `curl -X POST http://localhost:3000/api/login -H "Content-Type: application/json" -d '{"username":"test","password":"test"}'`. Check the response status and body. If 401, the credentials are wrong or the user does not exist.

- **Diagnose:**
  1. **Do the test credentials exist in the target service's user database?** Contact the service owner and ask: "Are the user 'test' / password 'test' (or equivalent) provisioned in the service?" If not, the database is not seeded with test data.
  2. **Are the credentials hardcoded in the k6 script vs. provided via environment variable?** In the script, search for `username: "test"` or similar. If hardcoded and the service uses different credentials, that is the mismatch. If the credentials are read from `__ENV`, confirm that the environment variables are set in the execution environment (e.g., `export TEST_USER=...`).
  3. **Has the service's user schema or auth mechanism changed?** Ask: "Was a recent change made to the login endpoint (e.g., OAuth, SAML, multi-factor authentication)?" If so, the k6 script may be using an obsolete auth flow.

- **Mitigate:**
  1. **Seed the test database with valid credentials:** Ask the service owner to either (a) add the test user via a database migration or API, or (b) provide a script that sets up test data (e.g., `./scripts/seed-test-db.sh`). Run the script before the load test.
  2. **Update the k6 script:** If credentials are hardcoded, change them to match the actual test user provisioned in the service. If credentials are environment variables, confirm they are set before running k6: `echo $TEST_USER`.
  3. **Use a privileged setup user:** If the test user does not have permission to perform the test actions (e.g., checkout), ask the service owner to provide a dedicated load-test service account with admin or elevated permissions.

- **Verify:**
  1. Run `curl -X POST http://localhost:3000/api/login -H "Content-Type: application/json" -d '{"username":"<new_user>","password":"<new_password>"}' and confirm the response status is 200 or 201.
  2. Run the k6 smoke test with the updated credentials: `k6 run --vus 1 --duration 2m load-test.js`. The setup phase should log "login successful" or similar, not "running unauthenticated".

- **Escalate if:**
  - The service does not have a way to provision test users (no seeding script, no API to create users).
  - The auth mechanism has changed and the k6 script cannot be easily updated (e.g., OAuth flow requires browser interaction).
  - **Escalate to:** Service owner / QA team for coordination of test data provisioning and script updates.

## 18. Post-Incident Review (PIR)

### What went well

- **Test framework was well-structured and caught the failure quickly.** The k6 script included an `abortOnFail` threshold that prevented wasted effort on a failing service; the test halted after 2 seconds instead of running the full 10-minute profile on a broken target.
- **Clear metric instrumentation.** The script defined custom metrics (error_rate_post_api_login, duration_post_api_login, etc.) and thresholds that made the failure obvious and reproducible.
- **Explicit risk classification in the script comments.** The script marked POST /api/login as HIGH-risk, which helped focus diagnosis on the authentication layer.

### What went poorly

- **No pre-flight service health check.** The test was executed against localhost:3000 without first verifying that the service was running and responsive. A 10-second smoke test (1 VU, all endpoints, 2xx check) before the main load test would have caught the issue immediately.
- **Silent failure mode in setup phase.** The setup function logged "login returned 500; running unauthenticated" and continued, masking the underlying service issue. The full test profile was never executed because of a threshold abort, but a simpler issue (service down) was disguised by defensive logging.
- **No centralized application logging.** The root cause of the 500 error is unknown because the service does not emit structured logs to a centralized aggregator. Diagnosis required manual SSH into the host and log file inspection (not scalable or auditable).
- **No monitoring or alerting on the service.** The service failure was discovered during a load test, not by automated health checks. Production users may have been affected before the load test revealed the issue.

### Action items

1. **Implement mandatory smoke-test pre-flight check before every load test.** One VU, all 4 endpoints, all responses must be 2xx. Fail the test run (abort, do not continue) if any endpoint returns non-2xx. **Owner:** Test executor / QA framework. **Due date:** +1 week.

2. **Add centralized structured logging (JSON format) to the service with trace ID, endpoint, HTTP status, latency, and error message for all 4xx/5xx responses.** Ship logs to a centralized aggregator (ELK, Splunk, or Datadog). **Owner:** Service owner. **Due date:** +2 weeks.

3. **Deploy continuous synthetic health monitoring (24/7, every 5 minutes).** Hit all 4 test endpoints and alert (page on-call) if error rate > 5% or latency p95 > 2× baseline. **Owner:** Platform team. **Due date:** +2 weeks.

4. **Instrument the service with Prometheus metrics:** export auth latency (p50/p95/p99), dependency health (availability %), and resource usage (CPU %, memory %). Create Grafana dashboards. **Owner:** Service owner / platform team. **Due date:** +3 weeks.

5. **Update the k6 test script: modify the setup function to abort the entire test run (not just log a warning) if login returns non-2xx.** This prevents masking of service issues. **Owner:** Test executor / QA. **Due date:** +1 week.

6. **Root-cause analysis of HTTP 500 on POST /api/login (if not already resolved by the immediate actions in §11).** Diagnose the exact exception or misconfiguration causing the error. File a P1 bug if the root cause is an application bug. **Owner:** Service owner / development team. **Due date:** +3 days.
